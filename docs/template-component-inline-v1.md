# Template Component Inline V1

DeepPrint 的模板生成路线调整为：组件库只作为生成素材，不作为最终 TemplateBundle 的运行时依赖。

最终用户模板应该尽量简单：

```text
manifest.json
template.typ
data.json
data.schema.json
```

不要在最终 Bundle 中要求用户理解或维护 `lib/` 目录。

## 1. 目标

让 AI 生成更稳定，但不把组件系统复杂度暴露给用户。

这里的 starter 必须很薄：它只提供页面物理框架、默认字体/边距、基础安全容器和最小示例数据，不预绑定完整组件集合。

DeepPrint 内部可以维护高质量 Typst 组件；AI 生成模板时可以借这些组件。但保存到用户模板里的结果应该是一个自包含 `template.typ`，拿给 `typst-json-render` 就能直接编译。

```text
DeepPrint 内置组件源码
-> AI/生成流程选择合适组件
-> 将需要的组件代码内联到 template.typ
-> 保存 template.typ + data.json + data.schema.json
-> typst-json-render 直接编译
```

## 2. 为什么不把 lib 放进模板

不推荐最终模板长这样：

```text
template.typ
data.json
data.schema.json
lib/
  core-v1.typ
  receipt-v1.typ
```

原因：

- 用户只想改一个模板，不想理解组件依赖
- AI 容易改坏 import 路径或组件边界
- 导出模板时文件更多，心智负担更高
- 组件升级和旧模板兼容会变成额外产品问题

V1 先接受少量代码重复，换更稳定的用户体验。

## 3. 组件库定位

组件库是 DeepPrint 的生成素材库，不是 TemplateBundle 的运行时包管理器。

推荐目录：

```text
template-assets/
  components/
    receipt-v1.typ
    shipping-label-v1.typ
    exam-paper-v1.typ
    business-document-v1.typ
    invitation-v1.typ
  starters/
    receipt-basic/
      template.typ
      data.json
      data.schema.json
      manifest.json
      design.md
    invitation-basic/
      ...
```

`components/*.typ` 只给 DeepPrint 和 AI 使用。用户模板不直接引用这些文件。
`starters/` 只提供页面尺寸、基础容器、默认风格、最小示例和同类型设计约束，不负责声明完整组件依赖。

第一版不要把组件库拆得过细。很多领域一个文件就能装下主要能力。先做“每个领域一个组件文件”，等以后文件真的过大、维护痛了，再拆成更小颗粒。

## 4. 最终模板形态

最终 `template.typ` 可以包含三段：

```typst
// Generated by DeepPrint.
// Component source: receipt-v1

// 1. inlined component helpers
#let safe-text(body, size: 8pt, weight: 400) = {
  box(width: 100%, clip: true)[#text(size: size, weight: weight)[#body]]
}

#let receipt-item-row(name, qty, amount) = {
  grid(
    columns: (1fr, 8mm, 16mm),
    safe-text(name),
    align(center)[#qty],
    align(right)[#amount],
  )
}

// 2. document entry
#let data = json("data.json")

// 3. render
#receipt-document(data)
```

用户看到的是一个可编辑的 `template.typ`，不是一组需要维护的模块。

## 5. 生成流程

```text
用户选择模板类型或自然语言描述
-> DeepPrint 选择 starter
-> DeepPrint 同时提供同领域 component source 和 design brief
-> AI 优先基于 starter + component source + design brief 排版
-> 必要时少量补写 Typst
-> 内联组件代码到 template.typ
-> AI 修改 template.typ / data.json / data.schema.json
-> typst-json-render 编译 PNG
-> 成功后保存 Bundle files_json
```

如果 AI 要做小票，它不需要发明 `receipt-document`，而是从 DeepPrint 提供的 starter 和同领域 component source 中继续改。AI 的工作重点变成：

- 调整字段和示例数据
- 复用并内联需要的组件模式
- 调整少量样式变量
- 调整区块顺序
- 在缺口处补充少量 Typst 代码

不是从零写整套页面结构。

## 6. 组件注释规范

组件注释本身就是 AI 使用说明的一部分。

不要只依赖系统 prompt。每个组件定义前都应写短注释，直接告诉 AI：

- 这个组件的用途
- 输入是什么
- 适合在什么场景用
- 不适合在什么场景用
- 有哪些排版边界不能破坏

推荐格式：

```typst
// receipt-item-row
// Purpose: single line item for a 58mm receipt.
// Inputs: name, qty, amount.
// Use when: rendering the main item list.
// Avoid: long notes, multi-line instructions, footer text.
// Rules: keep amount right aligned; keep label text clipped.
#let receipt-item-row(name, qty, amount) = { ... }
```

这类注释要短、硬、贴着定义写。它们既是组件说明，也是在代码旁边给 AI 的局部提示词。

## 7. AI 约束

Prompt 规则要短而硬：

```text
You are editing a self-contained TemplateBundle.
Use the provided starter as the base layout.
Use the provided component source as the primary reference for domain-specific layout and helpers.
Prefer adapting or inlining existing starter/component patterns instead of inventing new Typst structure.
Only write new Typst from scratch when the starter and provided component source do not cover the required layout.
When writing new Typst, keep it minimal and consistent with the existing starter/component style.
Do not add local lib import statements.
Only the allowlisted Typst package import `@preview/tiaoma:0.3.0` is allowed for barcode/QR.
Do not create lib/ files.
Keep all required Typst helpers inside template.typ.
data.json should only contain business data that changes per render.
Keep fixed labels, static headings, and decorative text inside template.typ.
Keep data.json and data.schema.json consistent.
Do not output a schema like {"type":"object","additionalProperties":true}.
```

中文产品描述：

```text
你正在编辑一个自包含模板。
必须以提供的 starter 作为基础版式。
必须优先参考提供的组件源码来完成领域排版和 helper 设计。
优先改造、内联 starter 和组件中的现有模式，不要随意发明全新的 Typst 结构。
只有当 starter 和组件源码都无法覆盖目标版式时，才允许少量手写 Typst。
手写 Typst 时要尽量少，并保持与 starter / 组件风格一致。
不要新增本地 lib import。
条码、二维码只能使用已 allowlist 的 `@preview/tiaoma:0.3.0`。
不要新增 lib/ 目录。
所有必要的 Typst helper 都必须保留在 template.typ 内。
data.json 只放每次渲染会变化的业务数据。
固定标题、静态标签、装饰文案保留在 template.typ 中。
data.json 与 data.schema.json 必须一致。
data.schema.json 不能是空壳 schema。
```

Starter 和组件注释也要重复这些规则。不要只靠系统 prompt。

## 8. Data 与 Schema 约束

保留 `data.schema.json`。它是数据契约，不是装饰。

V1 先不增加额外的 `schema quality check`。先依赖三层约束：

- starter 提供正确粒度的 `data.json` / `data.schema.json` 示例
- 组件注释明确输入字段和禁止项
- prompt 明确 `data.json` 只放业务数据，不放固定文案

`data.json` 应只包含“同一模板多次渲染时会变化的内容”，例如：

- 门店名
- 商品明细
- 金额
- 时间
- 订单号
- 二维码内容

以下内容默认留在 `template.typ`，不应该进入 `data.json`：

- 固定栏目名
- 静态标题
- 默认感谢语
- 装饰文本
- 分隔符文案
- 货币符号

`data.schema.json` 必须描述实际的 `data.json` 结构，不允许故意写成宽泛空壳 schema。

## 9. 迁移现有资产

`typst-json-render/bundles/*-basic` 里的 Typst 组件可以作为第一批素材：

- `receipt-basic/lib/receipt-v1.typ`
- `shipping-label-basic/lib/shipping-label-v1.typ`
- `exam-paper-basic/lib/exam-paper-v1.typ`
- `business-document-basic/lib/business-document-v1.typ`

迁移时不要照搬 `lib/` 运行时结构，而是拆成：

- DeepPrint 内置 component source
- DeepPrint 薄 starter
- 生成时内联后的 self-contained `template.typ`

## 10. 取舍

V1 接受：

- 多个模板重复一份 helper 代码
- 旧模板不会自动吃到组件库升级
- `template.typ` 会比纯业务代码更长

换来：

- 导出模板简单
- AI 不处理文件依赖
- 用户只理解一个模板文件
- `typst-json-render` 编译路径更稳定

如果未来模板数量很大、重复代码真的成为问题，再考虑显式组件包或库引用。现在不做。
